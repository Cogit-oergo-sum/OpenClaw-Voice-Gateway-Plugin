# Fast Agent V3.2 架构升级开发计划

## 背景与目标
在目前的 `fast-agent-v3.ts` 实现中，主要采取的是 **“绝对并联与竞速 (Racing & Parallel)”** 的架构。即用户输入后，SLC 立即不假思索地进行垫词回复，同时 SLE 并行拉起去匹配和调用工具。

根据最新的架构要求：
> “只要是 AI 要回复用户，都一定是 SLC 来说话。对于用户提问，需要 **同步等待 SLE 判断是否需要调用工具**。若需要，则通过守护进程触发 SLC 垫词；若不需要，则直接回答本轮。”
> “SLE 仅仅作为严谨的执行任务职责。”

现状与目标的 GAP：
1. **现状是并联盲答，目标是基于判定分流**。当前 SLC 是独立直接回复。我们需要改成：先由小模型/快速链作 Router 判断，或者通过 SLE 快速预判，然后再决定流向。
2. **现状 Watchdog 仅作后台兜底，目标是由 Watchdog/流程引擎 驱动 SLC 的垫词**。
3. **SLE 的前置处理缺位**。目前开始对话时，并没有强制 SLE 提取环境总结并赋给 SLC，也没有处理 ASR 热词、上下文压缩等专门流程。

为满足这一要求，我们需要在 V3.2 对核心控制流进行重构。

## 改造内容与实施路线

### 阶段一：流控逻辑重构 (FastAgentV3.process)
**目标**：将当前的 并行执行 改为 `SLE 工具判定 -> 策略分流 -> 执行与回复` 的串行/协同流。

1. **引入 Fast Router (工具意图预判)**:
   - 在接收到用户消息时，不再立即无脑调用 `this.slc.run`。
   - 增加一个快速大模型调用（或利用 SLE 模型的一段精简 prompt）进行 `Intent Detection`，同步返回一个 Boolean：`needs_tool_call`。
   - **时间预算控制**：这步判定必须在 `300ms` 内完成，以保证整体延迟。

2. **路由分流逻辑**:
   - **分支 A (needs_tool_call == true)**：
     - 拉起 SLE 开始调用具体工具和执行漫长流程。
     - **同步触发** Watchdog 向 SLC 发出特定的 Inject 信号，使 SLC 基于刚判断出的工具意图，说一句带有潜意识的回复（例如：“关于您的会议安排，我需要查阅一下系统...”）。
   - **分支 B (needs_tool_call == false)**：
     - SLE 本轮不工作，不调用任何工具。
     - 直接拉起 SLC 读取上下文、规则和画布执行最终回复。

### 阶段二：强化 SLE “严谨的执行任务” 职责
**目标**：剥离所有直接面对用户的情绪推演，让 SLE 专注于 JSON 进出与后台任务。

1. **画布总结强管控 (`CanvasManager` 优化)**：
   - 强制 SLE 在工具链结束时，产出高密度的 `summary` 字段，而非大段自然语言。
   - 去除 SLE 内关于人设语气 (Core Soul) 的提示词要求，只给它发“工具列表和数据结构”。
2. **初始化与守护任务 (`WatchdogService` 触发扩充)**：
   - 当收到会话开始 (Session Start) 事件，Watchdog 派发一个内置任务给 SLE，要求 SLE 获取环境变量（如天气、位置），并将 summary 写入 Canvas，然后再构造 SLC 的 Initial Prompt。

## 阶段验收标准 (已全部达成)
- [x] 测试用例涵盖：用户问纯聊天问题，系统能在极短时间内直接 SLC 回复，期间控制台没有任何 Tool 调用的日志。
- [x] 测试用例涵盖：用户问查天气，系统停顿几百毫秒后（SLE 判定完毕），SLC 优雅地抛出一句简短的提示让其稍等，并且后台确切地启动了工具链，并在获得画布结果后，Watchdog 发起二次无缝结合播报。
- [x] 所有的 AI 说话内容记录中，均来自于 `SLCEngine`，`SLEEngine` 不再直接推流任何文本给前端 TTS。

## 避坑与沉淀记录 (Lessons Learned - V3.2 调试总结)
1. **SLE 的异步非阻塞改造**：
   - **问题**：在 HTTP SSE (Server-Sent Events) 环境中，如果 `FastAgentV3.process` 内部使用 `await slePromise` 同步等待后台工具执行，会导致 `text-chat` 请求被挂起，不但直接阻断了客户端接受事件，而且还会阻断 Watchdog 基于 Memory 的异步事件推送。
   - **解法**：必须让 SLE 的执行 `catch` 转移到后台（即 `slePromise.catch`），释放主请求完成 `onChunk` 的回吐。
2. **Watchdog 的状态机唤醒要求**：
   - **问题**：Watchdog 内部的心跳触发 `__IDLE_TRIGGER__` （例如 15 秒静默唤醒）严苛依赖于 `status === 'READY' && is_delivered === true`。但在初始化时只给了 `PENDING`，导致挂机唤醒永久失效。
   - **解法**：`initializeSession` 完成预总结后，必须显式地打上 `READY` 且 `is_delivered = true`，使环境从启动初始就成为可被打断或监控的状态。
   - 同理，在 SLE 工具快速同步返回（`!isTimeout`）时，也必须补齐 `READY` 和 `importance_score = 1.0` 使得 Watchdog 捕捉进度并抛出 `__INTERNAL_TRIGGER__`。
3. **取消 Racing 模式的历史残留阻断**：
   - **问题**：在原版竞速模式中，SLC 只要完成，就会发送 `signal.slcDone = true` 以中断后续动作。导致 V3.2 中 SLC 即便只是垫词，也会秒杀掉 SLE 刚发起的 stream 解析流程，结果是 OpenClaw 本地工具根本没有机会执行。
   - **解法**：移除 SLE Stream 读取流对 `signal.slcDone` 的依赖。只要需要执行工具，绝不能被 SLC 的发话时序打断。
4. **大模型回复多样性陷阱**：
   - **问题**：Prompt 中写明“比如：没问题这就为您查”，大模型会极其懒惰地每次都逐字输出这个示例。
   - **解法**：将限制词语改写为了规则描述（“根据任务类别优雅且多样化地给出一句简短提示”），去除 Literal String 例句。
5. **UI 测试颜色的直观区分**：
   - 为了方便监控 V3.2 各大特性的执行流缝合：
     - **蓝色** (`relay-user`)：您的输入。
     - **白色** (`relay-chat`)：直连模型闲聊。
     - **红色** (`relay-waiting`)：任务分发前的垫词响应（Task Padding）。
     - **紫色** (`relay-internal`)：工具端完成异步返回，Watchdog 接管结果后的正式汇报（Task Reporting）。
     - **粉色** (`relay-idle`)：用户挂机状态下的超时暖场（Heartbeat Waking）。
     - **深灰色斜体** (`relay-thought`)：推流暴露的潜意识设定，真实还原 AI 在处理任务时的“画外音”。
