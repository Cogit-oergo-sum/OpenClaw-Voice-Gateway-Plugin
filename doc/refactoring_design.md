# Voice Gateway 核心架构重构方案 (最终版)

在坚持“`npm install` 即用”的轻量级开源底线，并同时面对复杂的流媒体交互下，我们针对之前架构中的五大缺陷与三大后续拷问，制定了如下重构法则与兜底机制。

---

## 1. 重构背景与核心理念
*   **废弃重型依赖**：抛弃 Redis 等一切需要外部额外部署的数据库中间件，完全利用 Node.js 原生的单线程内存原子性+本地文件系统组合。
*   **回归本质**：不改变 OpenClaw 本来的心智法则（如 `memory/` 与 Webhook），不强加不符合开源插件体量的架构幻想，在资源约束与体验中间做最现实的取舍。

## 2. 状态与内存管理演进
*   **状态容器**：采用内存 Map 维护进行中的通话状态 (`CallState`)。
*   **防越权校验 (UUID Token)**：弃用危险的纯 UserId 识别。`start-call` 时网关下发 `controlToken` (UUID)，前后端后续交互凭此口令，拦截 API 遍历掐断攻击。
*   **本地异步延迟写盘**：废弃密集的进程互斥文件锁。对话全在内存 Buffer 游走。只在通话结束或定期（5 分钟）单点触发一次串行 `fs.appendFile`。这完全契合了 OpenClaw 主 Agent 的文件监听通讯标准。

## 3. 异常恢复与边界声明
**针对“重度失忆”与云端漏费的终极拷问：**
*   **重启“扫雷”机制 (Reboot GC)**：我们在插件 `register` 入口处（即网关启动的瞬间），**强制注入一段扫雷逻辑**：遍历并读取本地 `call_states.json` 中残留的（如果启用了定期备份状态快照的话）历史 `AgentInstanceId`，主动向 ZEGO 服务端发送一波 `DeleteAgentInstance`。
*   **ZEGO 通话自然兜底**：即便网关连残影都没存下，因为前端页面断开或者无音频流推入，ZEGO 服务端的 RTC 会存在默认的**空闲心跳超时（通常 5-15 分钟）**，超时后会自动断开并停止计费。这也是我们不搞重型中间件的托底底气。

## 4. 大模型延迟掩盖机制 (Filler Hack)
**针对 900ms 物理极限与“抢场/粘连”的终极拷问：**
*   **前置垫话流 (Filler Stream)**：遇到 `tool_call` 第一时间输出预设填充词（如："好的，我这就查一下"）。
*   **防止 TTS 灾难级抢话粘连的设计**：
    如果我们二次大模型响应极快（300ms 完事），ZEGO TTS 读垫话还没读完就接到了正式回复的 SSE 流，如何防止机器变成结巴？
    *   **停顿控制法 (SSML / 标点截流)**：在吐出的垫话结尾，强制注入一个强停顿的标点（如："好的，我这就查一下。" 注意**必须是句号+换行**），或者根据 ZEGO TTS 引擎是否支持注入 SSML 标签：`<break time="500ms"/>`。这在流媒体合并时会强迫合成引擎做自然换气。
    *   **垫话语义适配**：确保预流的垫话属于“万金油承接词”，无论后半句接什么都自然。不追求二次处理时复杂的网关端音频对齐，把断句完全甩给流自然结尾符。

## 5. 极轻量 ASR 动态别名纠偏与安全落地 (防文件并发损坏)
**针对热词“丢失”与“并发写坏文件”的终极拷问：**
*   **弃用外部请求**：决不在此极速黄金链路上去调用 ZEGO 缓慢的 `UpdateAgentInstance`。
*   **局部秒切**：识别到错词时，仅在内存写入 `{周洁纶: 周杰伦}`，下文 System Prompt 自动带上修正。
*   **安全的长效记忆沉淀 (独立 JSON 隔离)**：坚决抛弃暴力 Append 主配置 `USER.md` 的业余做法（规避与主 Agent 产生竞态条件进而损坏 Markdown 语法的致命灾难）。
    **正确做法**：将本通电话收集的别名，在挂断时合并读写到一个**插件专属的独立数据域**：`workspace/voice_plugin_aliases.json` 中。
    Node.js 中利用 `fs.promises` 读取此专属小文件（几 KB），内存中 Merge 后完整覆写回去。即便并发写，由于这是纯 JSON 且单点由网关管控，可以用轻量的 `npm install proper-lockfile` 或者简单的同步队列确保原子性。主 Agent 通过别的途径（如被赋予该路径的读取权限）侧面了解别名词库。

## 6. 从零信任开始：防御“建联风暴”与恶意扣费
**针对 /voice/start-call 敞口被刷满并发额度的终极拷问：**
*   **控制倒置**：的确，仅仅下发 UUID 防挂断是自欺欺人。要防“防刷单”，必须在进门前设卡。
*   **内置轻量级滑窗限流 (Rate Limiter)**：既然不能要求复杂的 API Gateway，那我们的 Node.js Plugin 需要在 `start-call` 接口**内置一个基于 IP/UserId 的高压限流器**。例如：单 IP 每分钟最多发起 3 次通话建立。
*   **并发槽位熔断 (Circuit Breaker)**：在配置表 `config.schema.json` 中强制要求用户配死 `maxConcurrentCalls` (例如 8)。`CallManager` 内部如果发现 `activeCalls.size >= 8`，对所有的 `start-call` 统统直接返回 429 Too Many Requests。连向 ZEGO 发起请求的机会都不给黑客！用极简单的两行代码守住云端余额的底线。
