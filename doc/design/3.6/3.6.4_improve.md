## V3.6.4 验证自省与修复方案 (Post-Verification Analysis & Fixes)

在验证 3.6.4 版本的过程中，发现了以下 5 个核心背离与稳定性问题，需要立即修正：

### 10. 冗余的人设注入 (Persona Redundancy)
*   **问题现象**：`.llm_requests.log:L5` 显示 SLC 提示词中同时包含了 `compact_persona` 和原始的 `IDENTITY.md/USER.md`，导致上下文极其臃肿。
*   **分析原因**：代码 `src/agent/prompt-assembler.ts:L50` 在拼装时，盲目 append 了 raw 模板文件，未判定 `personaSnapshot` 是否已包含这些信息。
*   **修复方案**：重构 `PromptAssembler.assemblePrompt`，实行“有快照即闭环”原则：若 `state.metadata.compact_persona` 存在，则**不再**注入 `identity.md` 和 `user.md`，仅保留快照和实时环境。

### 11. 潜意识状态粒度过粗 (Subconscious Task Differentiation)
*   **问题现象**：`"status": "PENDING"` (执行中) 和 `"COMPLETED"` (已完成) 解析到的潜意识提示词几乎一致，导致 AI 在不同阶段的口吻缺乏张力。
*   **分析原因**：`SLCEngine.run` 将所有后台回报统一标记为 `internal` 类型，调用了同一个 `buildShadowThought` 分支。
*   **修复方案**：
    *   在 `prompts.ts` 中新增 `progress` 和 `delivery` 两种潜意识类型。
    *   `PROGRESS_REPORT`: 对应“正在努力处理，保持耐心”。
    *   `RESULT_DELIVERY`: 对应“结果已出，准备以 Jarvis 的优雅姿态进行汇报”。

### 12. 潜意识内容污染 (Subconscious Pollution via Raw Tool Response)
*   **问题现象**：当 `source: Async-Result-Delivery` 且 `scenario: SLC_CHAT` 时，AI 的潜意识里塞满了原始的 Tool JSON (含 Token 统计、Prompt 长度等无用信息)。
*   **分析原因**：`AgentOrchestrator.ts:L104` 提取 summary 时优先使用了 `canvas.task_status.summary`（此处可能存储的是工具原始输出），并将其透传给了 SLC。
*   **修复方案**：
    *   **严格分层**：在渲染 SLC 潜意识时，强制遵循 `direct_response > extended_context > summary` 的优先级。
    *   **数据清洗**：如果最终命中的是 `summary` 且包含 JSON 特征，必须进行 `decant`（脱壳）处理，禁止将 raw JSON 喂给 SLC 的潜意识。

### 13. [ERROR] Request was aborted. (Race Condition & Timeout)
*   **问题现象**：高频交互下频繁出现请求被强行中止。
*   **分析原因**：
    1.  `src/agent/slc.ts:L108` 默认的 900ms 超时过于激进，在并发 CPU 波动时极易触发。
    2.  `FastAgentV3` 与 `AgentOrchestrator` 存在双重 lock 机制，逻辑重叠导致了请求被误杀。
*   **修复方案**：
    *   将 `slc` 默认超时放宽至 1500ms，并引入 `AbortSignal` 的优雅生命周期管理。
    *   清理 `fast-agent-v3.ts` 中的冗余 `lockedSessions`，统一由 `Orchestrator` 负责事务锁。

### 14. 任务播报未对齐 (Task Broadcast Mismatch)
*   **问题现象**：后台任务同步完成后，SLC 回复内容仍在使用旧的缓存片段，未及时消费 `direct_response`。
*   **分析原因**：在 `REFINING` (回复提炼/润色) 流程中，虽然代码消费了 `direct_response`，但在正常的 `Async-Result-Delivery` (后台回报) 链路中，如果 `SUMMARIZING` 场景产生的提纯输出 (`refinedSummary`) 延迟或解析失败，逻辑会降级并回退到原始 `summary`。

*   **修复方案**：确保在 `Async-Result-Delivery` 链路中，SLC 的消费对象强制绑定为 SLE 提纯后的 `refinedSummary`。

### 15. 数据字段与链路一致性审计 (Data Field & Pipeline Audit)
*   **问题核心**：`SLEEngine` (解析端) 与 `ResultSummarizer` (生产端) 的 JSON 协议不一致。
*   **分析原因**：
    1.  `SLEEngine:L100` 强制寻找 `.response` 字段，而 `ResultSummarizer:L93` (SUMMARIZING 场景) 输出的是 `.direct_response`。
    2.  `SLEEngine.run` 在解析 `SUMMARIZING` 结果后，仅将其作为 `output` 返回，**完全丢弃**了 LLM 环境感知的 `status` 和 `importance_score` 字段。
*   **修复方案**：
    *   **协议对齐**：统一 `SUMMARIZING` 场景下的 JSON 响应解析逻辑，使 `SLEEngine` 优先读取 `direct_response`。
    *   **持久化增强**：在 `AgentOrchestrator` 或 `SLEEngine` 内部，捕获 `SUMMARIZING` 场景的 JSON 结果并同步调用 `canvasManager.appendCanvasAudit`，将状态写回 Canvas。

### 16. 全链路术语原子化与精细化对齐 (Fine-grained Terminology Alignment)
*   **问题核心**：系统内部存在多套语义重复但名称不同的术语（如 `intentHint` vs `taskIntent`），增加了代码审计难度。
*   **分析原因**：跨模块开发时缺乏统一的词汇表，导致局部变量名与 Canvas 存储名脱节。
*   **修复方案**：
    *   **废除**：`refinedSummary`, `intentHint`, `taskIntent`, `deliveredText`, `agentsMd`。
    *   **统一对齐表格**：
        | 逻辑定义 | Canvas 存储名 | 局部变量名 (Proposed) | Payload 参数名 (Proposed) |
        | :--- | :--- | :--- | :--- |
        | **技术原始输出** | `summary` | `rawOutput` | `raw_output` |
        | **人话提纯对白** | `direct_response` | `directResponse` | `direct_response` |
