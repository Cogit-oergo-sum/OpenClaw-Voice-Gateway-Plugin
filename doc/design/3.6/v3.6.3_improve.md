目前的问题与改进方案：

### 1. ROUTING 场景 User 角色输入冗余
- **描述**: "scenario": "ROUTING" 的 "role": "user" 不需要 ‘时间: 2026/3/25 04:23:42;’ 信息，因为已经在画布里了。
- **原因分析**: `SLEPayloadAssembler` 在组装 `ROUTING` 负载时，为了保证环境感知，硬编码注入了当前系统时间，但在 V3.6.2 中，User 角色已经包含了完整的 `Canvas Snapshot`，其中 `env.time` 已涵盖此信息，造成 Token 浪费和语义冗余。
- **解决方案**: 修改 `src/agent/sle-payload-assembler.ts`，在 `ROUTING` 组装逻辑中移除 `[Context] 时间: ...` 前缀，直接使用 `[Input]` 或仅依赖 `Canvas` 注入。

### 2. DECIDING 场景 System 提示词过载与协议漂移
- **描述**: "scenario": "DECIDING" 的 "role": "system" 混入了大量的无用内容（如 `AGENTS.md` 的全量规则），且输出冗余。
- **原因分析**: 目前 `DECIDING` 场景误将全局的 `AGENTS.md` (交互准则) 注入到了 logic 专家的 System Prompt 中。逻辑专家只需要关注任务逻辑和工具协议，不需要关注社交/沉浸式规则。此外，协议要求仅返回 `thought` 和 `intent`，但模型有时仍返回 `response`。
- **解决方案**: 
    - 修改 `src/agent/prompts.ts` 中的 `SLE_ACTION_PROTOCOL`，剥离与逻辑决策无关的 `AGENTS.md` 内容。
    - 严格限定输出 JSON 结构，若 `intent` 存在，强行清空 `response` 字段以防 SLC 垫词冲突。

### 3. SLC 人设刷新 (REFINING) 触发频率过高
- **描述**: 出现了两次更新 SLC 的人设，一次 "callId": "global"，一次具体 callId，只需要真正发起互动时刷新一次。
- **原因分析**: 系统在会话初始化和每次关键状态变更时都尝试 `REFRESH_PERSONA`（通过 `AgentOrchestrator`）。`global` 级别的刷新通常由单例触发，导致与具体交互链路中的刷新重叠。
- **解决方案**: 在 `src/agent/agent-orchestrator.ts` 中引入防抖或原子锁机制。确保单次用户交互回合内，仅由当前会话 callId 触发一次人设缝合（Refining）。

### 4. SUMMARIZING 场景时间信息冗余
- **描述**: "source": "Task-Result-Sync"，"scenario": "SUMMARIZING" 不需要 ‘[Context] 时间: ...’。
- **原因分析**: 总结引擎在组装时复用了通用的 Context 注入模板。
- **解决方案**: 修改 `src/agent/result-summarizer.ts` (或对应的 Assembler)，精简 Summary 场景的 User Prompt，仅保留 `[任务意图]` 和 `[原始输出]`。

### 5. Watchdog 主动播报链路冗余
- **描述**: 不需要 `Watchdog-Heartbeat` -> `DECIDING` 的调用，浪费且低效。
- **原因分析**: 现行 Watchdog 逻辑在发现任务更新后，先走一遍 `DECIDING` 判定意图（即便结果已出），再触发 `SUMMARIZING` 和 `SLC_CHAT`。
- **解决方案**: 优化 `src/agent/watchdog.ts` 状态机。当 `source` 为 `Watchdog` 且任务状态为 `READY` 或 `COMPLETED` 时，直接路由至 `Task-Result-Sync` 总结流，跳过逻辑决策阶段。

### 6. delegate_openclaw 意图传递偏差
- **描述**: `intent` 参数内容应为“将天气注意事项记录到指定文件 0325checklist.md”，而非直接复刻用户原话。
- **原因分析**: SLE 引擎目前的 `auto-repair` 机制或逻辑推导过于“懒惰”，直接提取了用户的 Input 文本作为工具的描述参数，缺乏逻辑重构。
- **解决方案**: 优化 `src/agent/prompts.ts` 中的工具描述引导，要求 SLE 在调用代理工具时，必须将用户需求转化为“清晰、具备上下文、逻辑完整的动作描述”。

### 7. SUMMARIZING 缺乏持续性记录
- **描述**: 针对耗时较长、OpenClaw 返回多次结果的情况，没有找到连续的记录。
- **原因分析**: 当前 `ToolResultHandler` 采用的是“一次性摘要”模式，任务结束或初次同步后即停止心跳监听。
- **解决方案**: 在 `src/agent/tool-result-handler.ts` 中针对 `OPENCLAW_DELEGATE` 类型任务开启“增量摘要”模式。每当后台工具返回分段日志或进度更新时，根据 `canvas.version` 判定，自动追加 trigger 触发多次 `SUMMARIZING`。
