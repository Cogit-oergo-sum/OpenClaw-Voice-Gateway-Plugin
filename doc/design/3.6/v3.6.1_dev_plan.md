# OpenClaw RTC V3.6.1 - Payload & Role Optimization Plan

基于对 V3.6.0 的 Review 结果，本项目旨在修复 Payload 荷载过重、用户消息重复以及助手预填充（Pre-fill）缺失的问题，将架构完全对齐 V3.6 的“原子化、按需加载”初衷。

---

## 步骤 1：输出开发计划 (Development Phases)

| 阶段 | 核心任务 | 说明 | 验证手段 |
| :--- | :--- | :--- | :--- |
| **Phase 1: 路由 Payload 瘦身** | 将 ROUTING 阶段的 `agentsMd` 替换为 `skillsSummary` | 剥离 Safety/Heartbeat 等无关指令，仅保留技能摘要。解决 3.6.0 的主要上下文污染点。 | `verify_v3.6.1_p1.ts` |
| **Phase 2: DECIDING 角色去重与引导补全** | 1. 消除双重用户输入<br>2. 增加 Assistant 影子推理预填充 | 修复 `history` 尾部与 `text` 的冲突；在 Deciding 场景请求末尾引入 `thought` 引导模型先思考。 | `verify_v3.6.1_p2.ts` |
| **Phase 3: ASR 纠错协议标准化** | 统一使用 `prompts.ts` 中的 ASR 指令模版 | 消除 `sle-payload-assembler.ts` 中的硬编码字符串。 | `verify_v3.6.1_p3.ts` |
| **Phase 4: 整体回归测试与文档更新** | 运行全量验证脚本并更新 `overview.md` | 确保在原子化瘦身之后，模型意图识别准确率不下降，TTFT 优化 30%+。 | `verify_v3.6.1_final.ts` |

## 步骤 2：输出给 Coding Agent 的提示词 (Agent Prompts)

### 🤖 Coding Agent A: Payload 瘦身专家 (Phase 1 & 3)
**项目背景**：OpenClaw RTC V3.6.1 性能优化。
**核心参考**：`src/agent/sle-payload-assembler.ts`, `src/agent/prompts.ts`, `src/agent/fast-agent-v3.ts`。
**任务详述**：
1.  **修改 `SLEPayloadAssembler.assemble`**：
    - `ROUTING` 场景：传入的 `agentsMd` 参数应在 `FastAgentV3` 中被预处理为仅包含 `SkillRegistry` 注册的可用技能摘要，清理掉安全指令、管理准则等噪音 Token。
    - `ASR_CORRECTION` 场景：引用 `prompts.ts` 中的 `ASR_CORRECTION_DIRECTIVE_TEMPLATE` 替代原有的硬编码文本。
2.  **修改 `FastAgentV3`**：在调用 `assemble` 时，针对 `ROUTING` 场景，提取 `SkillRegistry` 的核心指令而非全量 `AGENTS.md`。
**强制闭环**：编写 `verify_v3.6.1_p1.ts`，导出组装后的 JSON Payload，验证 Token 长度是否显著下降且 ASR 模版正确。验证脚本必须显式调用 `process.exit(0)`。

---

### 🤖 Coding Agent B: 角色逻辑专家 (Phase 2)
**项目背景**：修复 V3.6.0 消息角色冲突与由于缺少预填充导致的幻觉风险。
**核心参考**：`src/agent/sle-payload-assembler.ts`, `src/agent/shadow-manager.ts`。
**任务详述**：
1.  **逻辑去重**：在 `DECIDING` 场景中，检测 `params.dialogueHistory` 的最后一个 `user` 角色消息是否与 `params.text` 内容一致。如果一致，则不再重复 push 新的 `user` 消息。
2.  **Assistant 预填充 (Shadow Thought)**：
    - 在 `DECIDING` 场景的消息序列结尾，追加一个 `role: 'assistant'` 消息。
    - 内容获取：通过 `buildShadowThought` 处理后的内容。
    - **预期布局**：... -> History (User) -> History (Assistant) -> User (Input) -> Assistant (Shadow Thought Pre-fill)。
**强制闭环**：编写 `verify_v3.6.1_p2.ts`，验证组装后的 Payload 最后一个消息是 `assistant` 角色，且无连续重复的 `user` 消息。验证脚本必须显式调用 `process.exit(0)`。

---

### 🤖 Coding Agent C: 回归与文档专家 (Phase 4)
**项目背景**：V3.6.1 核心逻辑优化完成后的整体集成验证。
**核心参考**：`overview.md`, `.llm_requests.log`, `doc/design/3.6/v3.6.1_dev_plan.md`。
**任务详述**：
1.  **全量回归验证**：编写 `verify_v3.6.1_final.ts`，顺序调用此前所有验证脚本（p1, p2, p3），确保整体链路在大压力、复杂语境下依然稳定。
2.  **Payload 瘦身审计**：清空 `.llm_requests.log`，通过模拟用户语音（如：查深圳天气），提取最新的 ROUTING 与 DECIDING Payload JSON，对比 V3.6.0 的 Token 消耗，计算优化率并记录在文档中。
3.  **更新项目全景 (Overview)**：更新 `overview.md` 中的架构描述，明确“分层 Payload 组装”与“Shadow Thought 预占推理”的技术规范。
**强制闭环**：输出 `v3.6.1_final_report.md`，包含核心验证数据与系统截图，并显式执行 `verify_v3.6.1_final.ts` 确保全绿通过。
