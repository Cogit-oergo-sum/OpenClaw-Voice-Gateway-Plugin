1. [x]当任务频繁更新的时候，太过于快速的同步进展，用户会跟不上。
    - 当任务状态为Complete、Failed时，播报
    - 当任务状态为Ready+重要程度中高时，通知用户。
    - 当任务状态为Pending时，只有任务重要程度高的时候，才需要处理并通知用户。
    - 其他都不播报
    - 从整体上，控制播报频率/间隔，不要太频繁
    - **[现状分析]**: 目前 `watchdog.ts` 中 `PENDING` 状态只要有 summary 变更就会触发播报，缺乏分级策略；且 `FAILED` 状态未在触发白名单中。
    - **[优化方案]**: 在 `WatchdogService` 中实现状态+分值双重过滤。设置 `PENDING_BROADCAST_THRESHOLD = 8` (高)，`READY_BROADCAST_THRESHOLD = 5` (中)，`COMPLETED/FAILED` 门槛为 1。
2. [x]当用户询问之前任务结果时，意图识别判断不需要天气，但是需要把`isAnswerInCanvas`输出为true，否则回答错误
3. [x]需要控制 SLC的提示词的最后增加 TTS_FRIENDLY_PROTOCOL。
    - **[现状分析]**: `PromptAssembler.assemblePrompt` 目前仅拼接了人设和时间，缺乏对语音输出格式（去标签、字数控制）的强制约束。
    - **[优化方案]**: 在 `PromptAssembler` 输出 SLC Prompt 时，尾部追加 `TTS_FRIENDLY_PROTOCOL`，确保模型不输出 Markdown。

6. 为啥Async-Result-Deliver，会有两次，一次SUMMARIZING，一次SUMMARIZING_RESULT
    - **[现状分析]**: 这是由于 `sle.ts` 在执行 `SUMMARIZING` 场景时，分别记录了 `Request+RawOutput` 和 `ParsedResult`。
    - **[优化方案]**: 现状合理，仅为审计日志服务，并非冗余 LLM 调用，无需修改代码逻辑，可在查看日志时忽略。
7. [x]目前"scenario": "DECIDING"包含的轮次太多了，要减少到5轮对话
    - **[现状分析]**: `AgentOrchestrator` 默认读取 20 轮消息作为全量 context，增大了 `DECIDING` 场景的 Token 消耗和干扰。
    - **[优化方案]**: 在 `PromptAssembler.assembleSLEPayload` 中针对核心决策场景对 `dialogueHistory` 执行 `slice(-5)` 操作。
8. [x] 针对 openClaw 返回的内容，仅查看 payloads 中的内容。
    - **[现状分析]**: `TaskDelegateSkill` 曾直接透传 `stdout` 或仅提取 `payloads[0]`，导致 SUMMARIZING 场景接收到大量冗余 JSON 噪音，干扰模型判断。
    - **[优化方案]**: 在 `DelegateExecutor` 中增加 `distill` 静态方法，统一从 `parsedData` 的 `payloads` 数组中提取所有 `text/content` 并合并，过滤掉原始 JSON 和 STDOUT 噪音。已在 `TaskDelegateSkill` 和 `weather_mcp` 桥接中应用。
