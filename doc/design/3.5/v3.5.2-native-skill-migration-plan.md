# Fast Agent V3.5.2：声明式原生技能 (Native Declarative Skills) 迁移计划

在 V3.5 中，我们已经实现了声明式外挂技能的动态加载（基于 `skills_repo` 目录结构）。然而为了安全和平稳过渡，我们的底座保留了“双轨制”——即 `TaskDelegateSkill` 和 `AsrCorrectionSkill` 依然是通过硬编码在 `src/agent/skills/core/` 并使用 `registerCoreSkills()` 强制绑定的。

按照 3.5 规划的终局形态，我们需要完成**“大一统”**：所有技能（哪怕是原生高速核心能力）都应拔除 TS 硬编码，转而使用最新的 `SKILL.md` （基于 YAML Frontmatter）形式声明和挂载。

---

## 一、改造目标与设计难点

### 目标 (Goal)
1. **删除** `src/agent/skills/core/*.ts` 下的冗余硬编码类结构。
2. **统一注册**：去除 `SkillRegistry` 的双轨制，仅保留 `loadFromDirectory()`，一切能力基于配置盘扫描。
3. **隔离无状态配置与有状态逻辑**：使 LLM Schema 定义彻底声明式化，而将底层真正的内存调用变成通用的分发处理器。

### 难点 (Challenge)
现有的外部 MCP 工具通常通过 HTTP 请求 (`method: POST`, `endpoint: http://...`) 与外界交互。然而，`asr_correction` 需要操作系统内部上下文（`CallManager.aliasMap`），`delegate_openclaw` 需要高强度操作底层进程及内存缓冲区 (`DelegateExecutor`, `CanvasManager`)。它们不能简单地替换成脱离当前 Node.js 进程的 HTTP 请求。

---

## 二、架构演进方案：Runtime 策略路由

我们将在 `SKILL.md` 中引入 `runtime`（执行引擎）的概念。
- 当 `runtime: mcp`（或 http）时，由 Wrapper 发起网络调用。
- 当 `runtime: native` 时，由 Wrapper 将参数通过事件或依赖注入投递给系统内部预先注册的守护对象进行处理。

---

## 三、三阶段落地计划

### Phase 1: 建立原生技能配置库 (Declarative Repository)
在底座外挂盘 `skills_repo/` 下，为原生核心补充声明配置：

**1. `skills_repo/core_delegate/SKILL.md`**
```yaml
---
name: delegate_openclaw
description: "委派复杂任务给主控助理。当需要执行查询、操作、检索等任何外部长效型任务时调用。"
runtime: "native"
isLongRunning: true
parameters:
  type: object
  properties:
    intent:
      type: string
      description: "委派意图的具体内容"
  required:
    - intent
---
[技能使用说明与文档：这是原生的本地工具链。]
```

**2. `skills_repo/core_asr_correction/SKILL.md`**
```yaml
---
name: correct_asr_hotword
description: "纠正 ASR 识别出的错误同音词并将其推入热词权重，防范持续的系统级幻觉。"
runtime: "native"
parameters:
  type: object
  properties:
    wrong: 
      type: string
      description: "ASR 听错的原始错别词（同音词）"
    correct: 
      type: string
      description: "推论出的正确表达（真词）"
  required:
    - wrong
    - correct
---
[技能使用说明与文档：这是原生的拦截与内存纠偏机制，必须静默极速工作。]
```

### Phase 2: 改造 DynamicSkillWrapper 支持本地驱动 (Native Binding)
由于所有解析均交由 `DynamicSkillWrapper` 完成，我们需要为它扩充 Native 的执行链路：
1. **解除硬编码依赖**：重构 `SkillRegistry`，在初始化时允许传入或保留注入内部依赖（如 `executor`, `summarizer`, `callManager` 等底座实例）。
2. **执行路由分发**：
   修改 `DynamicSkillWrapper.execute()` 逻辑。
   判断 `this.config.runtime`：
   - 如果是 `native`：通过预先在底座中注册的一个字典 `nativeHandlers.get(this.name)` 捞取到闭包函数。直接执行原逻辑（在 `TaskDelegateSkill` / `AsrCorrectionSkill` 中的业务执行代码转移到这里的句柄函数中）。
   - 如果是 `mcp/http`：保留之前的发送网络请求。

### Phase 3: 清理旧纪元包袱 (Unify Registry)
1. **删除大跃进包袱**：可以直接干掉 `src/agent/skills/core` 文件夹中的 `asr-correction.ts` 与 `task-delegate.ts`。
2. **清除双轨制**：在 `tool-result-handler.ts` 中，彻底移除 `registry.registerCoreSkills(...)` 这段缝合线代码。
3. 把系统原本对 `delegate_openclaw` 的依赖，转换为依赖初始化时的 `Native Handler` 注册：
   ```typescript
   // ToolResultHandler 初始化或 Registry 初始化时：
   const registry = SkillRegistry.getInstance();
   registry.registerNativeHandler('delegate_openclaw', async (args, callId, manager) => { ...原有 task-delegate 逻辑... });
   registry.registerNativeHandler('correct_asr_hotword', async (args, callId, manager) => { ...原有 asr-correction 逻辑... });
   // 然后一把扫盘，直接读取 schema 完成所有挂载
   registry.loadFromDirectory(repoDir);
   ```

## 四、安全测试基准 (Validation)
任何对大动脉的迁移绝不能流血，你必须通过以下全量原生地狱测试：

1. **核心测试通道 (ASR & Intent)**: `npx tsx verify_asr_v3.4.ts`
   - **断言（Assert）**：必须如丝般顺滑通过。ASR 同音词幻觉能够正确被 `SKILL.md` 识别并挂载给大模型，纠错词权重自动叠加，证明 Native Binding 路由正确！
2. **运行时检查**：启动进程 (`./ctl.sh restart`) 能够确保 Console 日志中统一打出：`[SkillRegistry] 已动态加载挂载技能: delegate_openclaw (runtime: native)` 等字眼，而系统内再无硬编码。

### Phase 2: 改造 DynamicSkillWrapper 支持 Native Binding
```markdown
# 任务目标: V3.5.2 第二阶段 - Wrapper 的 Runtime 策略路由与 Registry 原生绑定

## 1. 项目背景
在上一个阶段，我们将原本硬编码的原生技能转化为了 \`skills_repo\` 下的声明式 \`SKILL.md\` 文件。但这些衍生出的 \`DynamicSkillWrapper\` 当前只会做 MCP/HTTP 调用。我们需要改造它使之支持 \`runtime: native\`，直接调用 Node.js 内存里的闭包句柄，以实现内部核心的高速调度。

## 2. 需参考核心文档/代码
- 外壳加载与执行器: \`src/agent/skills/DynamicSkillWrapper.ts\`
- 技能中心与挂载点: \`src/agent/skills/index.ts\`（SkillRegistry)

## 3. 开发详细计划
1. 设计挂载容器：在 \`SkillRegistry\` 中新增一个私有变量 \`nativeHandlers = new Map<string, Function>()\`，并提供一个对外 API \`registerNativeHandler(name: string, handler: Function)\`。
2. 修复声明加载：修改 \`DynamicSkillWrapper\` 的解析构造函数，保证能通过 \`yaml.runtime\` 读取到 \`native\` 等属性。如果没提供默认为 \`mcp\`。
3. 执行器路由分支：修改 \`DynamicSkillWrapper.execute()\` 方法。若 \`runtime === 'native'\`，则向 \`SkillRegistry.getInstance()\` 查询是否存在对应的 Native Handler。若存在，直接将 args、callId、canvasManager 等核心参数样投递给闭包函数，并 \`await\` 返回代理执行的结果。

## 4. 单元测试要求
你需要针对新建立的架构编写简单的插桩调用验证：保证通过 \`registerNativeHandler('test', ...)\` 绑定并在大盘扫到配置后，调用其 \`execute\` 会准确跳过 HTTP 路线走本地路由回调且返回对应结果。
```
