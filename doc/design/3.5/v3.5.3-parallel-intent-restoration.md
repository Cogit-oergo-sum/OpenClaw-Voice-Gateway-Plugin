# Fast Agent V3.5.3：原生并行意图识别与工具链恢复设计文档

## 1. 需求背景 (Problem Statement)

### 1.1 现状与病灶
在 Fast Agent V3.3/V3.4 的演进中，为了追求极速的 TTFT（首包延迟），我们引入了 `IntentRouter`。目前的逻辑存在以下核心问题：
- **独裁式路由**：`IntentRouter` 判定 `needsTool: true` 后，会强迫 SLE 引擎必须选出一个意图并执行工具。这导致了 **“指鹿为马”**——当用户进行 ASR 纠错（如“不是即刻是极客”）时，由于包含某些动词，被误判为业务指令并强行挂载了业务工具。
- **排他性逻辑**：当前的路由是二进制的（Task 或 Chat），不支持复合意图。当用户说出“不是机构科技是极客，另外查一下极客科技的待办数据”时，系统只能处理其中一个，且极易误判。
- **非标准 Skill 污染**：为了区分耗时，曾考虑引入私有的 `isLongRunning` 标记，这破坏了与 OpenClaw / Claude Code 生态中标准的 `skill.md` 规范的兼容度。

---

## 2. 处理方案 (The Solution: V3.5.3 Evolution)

### 2.1 权力下放：IntentRouter 降级为“垫词顾问”
- **核心职能**：`IntentRouter` 不再拥有对 SLE 决策的干扰权。它仅服务于 SLC（极速聊天引擎），告诉 SLC 用户意图是否包含**“看起来比较慢的高耗时动作”**。如果是，SLC 抢先输出过渡播报（Filler）；否则，SLC 保持静默或正常回复。
- **动态感知**：Router 的 Prompt 不再包含硬编码的规则（如“不要纠错”），而是由系统在运行时自动注入当前系统拥有的技能（Skills）及其工具属性，利用 LLM 的常识判定该工具是否为“高耗时外联操作”。

### 2.2 架构还原：启用原生 Parallel Tool Calling
- **决策回归**：将“什么时候调用工具”的决策权完全交还给 SLE 引擎及大模型原生的 **Function Calling / Parallel Tool Calling** 能力。
- **全量暴露**：SLE 在执行时，不再受 Router 的 `needsTool` 信号控制，始终同时暴露：
    - **Native Core Tools** (如 `asr_correction`)
    - **Business Delegate Tools** (如 OpenClaw CLI)
    - **Skill Repo Tools** (如 Weather, RAG)
- **复合处理**：利用大模型一次输出多个 Tool Call 的能力，实现一次请求内同时完成“ASR 纠错”与“业务查询”。

### 2.3 极致兼容：标准 Skill.md 支持
- **无感适配**：完全兼容标准的 `skill.md` 文件夹结构。不需要任何私有 YAML 字段。
- **自动推导**：通过 Router 探测 Skill 中包含的工具类型（例如是否包含 `delegate_openclaw` 或网络端点）来逻辑推导是否开启垫词，而非硬编码描述。

---

## 3. 架构对比 (Architectural Differences)

| 维度 | V3.3-V3.4 (现状) | V3.5.3 (本方案) |
| :--- | :--- | :--- |
| **判定逻辑** | Router 判定 `needsTool` 后，强行命令 SLE 执行工具。 | Router 仅触发 SLC 垫词；SLE 根据全量上下文自主抉择。 |
| **并发能力** | 单意图，ASR 纠错与业务命令互斥。 | 原生并行意图调用，支持纠错+查资料同步执行。 |
| **标准兼容** | 逐渐引入 `isLongRunning` 等私有冗余字段。 | 100% 兼容 OpenClaw / Claude Code 原生 `skill.md`。 |
| **扩展性** | 每加一个新工具，可能都要改一次 Router Prompt。 | 工具能力自动注入 Router 上下文，意图动态归位。 |

---

## 4. 实施计划 (Implementation Walkthrough)

### 第一阶段：Prompts 与路由机制改造
1. **[prompts.ts]**: 
    - 将 `INTENT_ROUTER_SYSTEM_PROMPT` 转为动态生成逻辑。
    - 移除 `SLE_ACTION_PROTOCOL` 中的“强迫执行”类威压指令。
2. **[intent-router.ts]**: 
    - 构建 `SkillContext` 注入逻辑，在调用 Router 前提取当前技能摘要。

### 第二阶段：SLE 并行调用恢复
1. **[sle.ts]**: 
    - 移除 `forceToolIntent` 开关，解除对工具列表（Tools Filter）的过滤。
    - 调整 Payload 以完全支持 Parallel Tool Calling。

### 第三阶段：结果处理链优化
1. **[tool-result-handler.ts]**: 
    - 确保 `handleToolCalls` 能无缝处理数组级工具返回，并配合 Canvas 状态进行复合展示。

## 5. 子 Agent 开发提示词 (Subagents Prompts)

### 第一阶段：Prompts 模板化与路由动态注入
```markdown
# 任务目标: V3.5.3 Phase 1 - 意图路由动态上下文注入

## 1. 项目背景
你正在维护 OpenClaw RTC Fast Agent (V3.5.3)。我们正在重构意图路由机制，使其从“强行分流”转为“动态感知”。核心目标是让 IntentRouter 能根据当前加载的 Skills 自动判定是否需要垫词。

## 2. 参考文档/代码
- 设计要求: `doc/design/3.5/v3.5.3-parallel-intent-restoration.md`
- 核心代码: `src/agent/prompts.ts`, `src/agent/intent-router.ts`, `src/agent/skills/index.ts`

## 3. 开发详细计划
1. **Prompts 重构**: 将 `prompts.ts` 中的 `INTENT_ROUTER_SYSTEM_PROMPT` 修改为一个动态生成的模板函数，能够接收“长耗时工具清单”作为字符串参数。
2. **路由注入**: 修改 `intent-router.ts` 中的 `detectIntent` 方法。在组装消息前，从 `SkillRegistry` 获取当前所有技能的摘要，并注入到判定准则中。
3. **判定调优**: 显式告诉模型：如果意图仅包含 ASR 纠错或简单理解（不涉及外部 Delegate 指令），禁止返回 needsTool: true。

## 4. 单元测试要求
- 编写测试脚本，模拟一个包含 `delegate_openclaw` 的技能和一个仅包含 `asr_correction` 的技能，验证 Router 在纠错时是否返回 false，在命令时是否返回 true。
```

### 第二阶段：SLE 决策权回归与并行工具调用
```markdown
# 任务目标: V3.5.3 Phase 2 - SLE 原生并行工具链恢复

## 1. 项目背景
剥夺 IntentRouter 的“独裁”地位。我们要让 SLE 引擎重新获得完整的意图决策权，并支持一次性纠正发音 + 执行任务的仿生能力。

## 2. 参考文档/代码
- 设计要求: `doc/design/3.5/v3.5.3-parallel-intent-restoration.md`
- 核心代码: `src/agent/sle.ts`, `src/agent/fast-agent-v3.ts`

## 3. 开发详细计划
1. **解除锁定**: 在 `sle.ts` 中删除所有关于 `forceToolIntent` 的逻辑。不应再通过系统指令强迫大模型必须从工具列表中选择执行。
2. **工具全暴露**: 无论 Router 返回什么，SLE 调用 LLM 时必须传入全量的已注册 Tools Schema。
3. **Parallel 支持**: 确保 API 调用参数允许并支持 Parallel Tool Calling。

## 4. 单元测试要求
- 终端执行：`npx tsx verify_asr_v3.4.ts` 进行零退化验证。
- 验证日志：输入闲聊内容时，SLE 不再出现“已判定当前需要执行工具”的幻觉指令。
```

### 第三阶段：多工具链式执行与复合语义验证
```markdown
# 任务目标: V3.5.3 Phase 3 - 复合意图执行验证与 Canvas 整合

## 1. 项目背景
最后冲刺。验证系统在面对“纠错 + 搜索”复合语句时的表现。确保多个 Tool Calls 的结果能顺畅地同步/异步写入 Canvas 并在 SLC 实现分级披露。

## 2. 参考文档/代码
- 设计要求: `doc/design/3.5/v3.5.3-parallel-intent-restoration.md`
- 核心代码: `src/agent/tool-result-handler.ts`, `src/agent/canvas-manager.ts`

## 3. 开发详细计划
1. **执行器鲁棒性**: 确保 `handleToolCalls` 在处理多个工具回包时，其 PENDING 状态标记和结果汇总不会互相覆盖。
2. **ASR 优先级确认**: 验证在复合调用中，`correct_asr_hotword` 作为一个 Native 同步工具，能被优先执行并生效。

## 4. 单元测试要求
- 编写集成测试 `verify_parallel_v3.5.3.ts`。
- 测试输入：“不是机构是极客，帮我查一下极客科技的待办”。
- 期望结果：控制台同时捕获到 ASR_CORRECTED 事件和 DELEGATE_EXECUTING 事件。判定成功。
```
