# V3.7.1 核心流程对齐与状态流转修正计划

## 1. 现状问题 (Problem Statement)
- **流程错位**: 当前 `AgentOrchestrator` 在处理 `__INTERNAL_TRIGGER__` (Watchdog 触发) 时才去调用 SLE 进行 `SUMMARIZING`。这会导致：
  1. 用户感知延迟增加（播报前需要额外等待一次 LLM 调用）。
  2. 职责越界：Orchestrator 承担了逻辑原子层的提纯工作。
- **状态不完整**: `ToolResultHandler` 在工具结束后直接将任务置为 `READY`，但此时 Canvas 中缺失精炼后的 `direct_response`。

## 2. 目标 (Objectives)
- **前置提纯**: 在工具执行结束的第一时间（`ToolResultHandler` 内部）完成 SLE 总结，实现“离线思考，在线播报”。
- **Orchestrator 瘦身**: 移除 `orchestrate` 中针对内部触发的逻辑重叠，使其回归纯粹的调度与 SLC 唤起。

## 3. 任务分解 (Breakdown)

### Phase 1: ToolResultHandler - 异步生命周期补完
- [ ] 修改 `handleToolCalls` 中的长耗时工具完成分支。
- [ ] 引入 `SLEEngine.run(..., 'SUMMARIZING', ...)` 的调用。
- [ ] 将提纯后的 `direct_response` 与原始 `summary` 一并写入 Canvas。
- [ ] 将任务状态切换为 `READY`。

### Phase 2: AgentOrchestrator - 清理越界代码
- [ ] 移除 `orchestrate` 方法中针对 `__INTERNAL_TRIGGER__` 的 `SLE (专家提纯汇报)` 逻辑块。
- [ ] 调整 `directResponse` 的聚合逻辑，直接从 `targetTasks` 提取已有的 `direct_response` 或 `summary`。
- [ ] 确保 `SLC` 接收到的 `tasks` 数组保持最新状态。

### Phase 3: 验证与回归
- [ ] 模拟工具执行完成，验证 Canvas 中是否同时生成了 `summary` 和 `direct_response`。
- [ ] 模拟 Watchdog 触发，验证 SLC 播报是否能够瞬间启动（无前置 SLE 延迟）。
- [ ] 运行 P0 回归测试。

## 4. 影响范围 (Impact)
- `src/agent/tool-result-handler.ts`: 增加了一次 SLE 调用。
- `src/agent/agent-orchestrator.ts`: 代码大幅精简，响应速度提升。
- `src/agent/sle.ts`: 无需修改，但其 `SUMMARIZING` 场景将被新路径调用。
