# 3.7 Multi-tasking (多任务并发与状态感知) - 设计全案 (Story & Design) [Status: IMPLEMENTED]

## 1. 核心体验故事：Alex 的办公桌前 (Jarvis on the Desk)

为了定义 3.7 版本的交互之魂，我们看一个名为 Alex 的用户与 Jarvis 在办公桌前的典型交互场景。这些任务均基于 openClaw 目前已有的工具集（文件读取、Shell 执行、天气查询等）。

### 阶段一：多任务齐发 (The Multi-intent Trigger)
**时间：09:00 AM**
**场景**：Alex 刚坐到办公桌前，对 Jarvis 下达指令：
> **Alex**: “Jarvis，帮我看下当前目录下最新的 `report.txt` 讲了啥，顺便把我 `Downloads` 文件夹里所有的 PDF 文档整理到一个叫 `Archived_PDFs` 的新文件夹里。”

**Jarvis 的反应**：
> **Jarvis**: “好的先生，正在为您读取汇报文件，并同步执行下载文件夹的清理与归档。请稍等，我正在整理信息。”

---

### 阶段二：并行中的“插话”与闲聊 (Deeply Parallel Soul)
**时间：09:02 AM**
**场景**：Alex 开始整理桌面，PDF 移动任务仍在后台执行（Shell 命令处理中）。
> **Alex**: “对了 Jarvis，上海今天下午会有雨吗？我打算四点左右出去见客户。”

**Jarvis 的反应**（即时秒回，同时不丢失后台任务）：
> **Jarvis**: “上海今天下午会有零星阵雨，气温 20 到 24 度，建议您带把伞。`report.txt` 已经读到了，总结已经在 Canvas 上。文件夹清理任务正在扫描，文件量略大，但我正在全力处理，请再给我几秒钟。”

---

### 阶段三：进度追问与意图对焦 (Proactive Precision)
**时间：09:04 AM**
**场景**：Alex 准备起身拿咖啡，还没收到文件夹整理完的消息。
> **Alex**: “文件夹整理还没弄好吗？”

**Jarvis 的反应**（精准进度自述）：
> **Jarvis**: “新文件夹 `Archived_PDFs` 已经建好了，我目前正在移动里面的 15 个文件。已经处理完一半了，再稍微等等。”

---

### 阶段四：优雅的汇总与闭环 (The Stitched Closing)
**时间：09:05 AM**
**场景**：Alex 拿着咖啡回来，所有任务全部完结。
> **Jarvis**（主动感应 Alex 停止说话后，优雅切入）：
> “先生，办妥了。15 个 PDF 文件已全部移动至 `Archived_PDFs` 目录。另外，`report.txt` 的核心内容是关于下周的项目节点，我已经为您提炼了摘要供您查阅。下午出门请记得带伞。”

---

### 阶段五：短效追问与静默归档 (Short-term Query & Silent Archiving)
**时间：09:06 AM**
**场景**：Alex 突然追问刚才播报提到的摘要细节。
> **Alex**: “等等，你刚才说报告里的下周项目节点，具体是哪几天？”

**Jarvis 的反应**（游刃有余，利用短效驻留或归档记忆）：
> **Jarvis**: “报告里提到的是周二的架构评审和周四的联调发布，先生。”

**时间：09:10 AM**
**场景**：Alex 开启全新话题，不经意间后台已完成 Canvas 清理。
> **后台逻辑**：系统判断任务已播报结束超过 3 轮对话（TTL 且 `is_delivered=true`），自动将 `report.txt` 的核心摘要和文件整理日志从 Canvas 热区移除，压缩存入长期记忆库（DialogueMemory），画布恢复干爽清白。

## 2. 闭环验证：场景 Payload 与 Message 快照 🌟

为了后续开发及 TDD 测试验证，以下定义了每个核心场景下 **SLC (交互层)** 与 **SLE (逻辑层)** 必备的消息上下文结构（Protocol）。

### 2.1 任务流式发起 (Intent Forking)
- **触发**：识别出两个及以上独立任务指令（读文件 + Shell 指令）。
- **当前 Canvas 物理状态**: `{"tasks": []}` (空)
- **步骤 1: SLE (ROUTING) 意图识别的 Message 上下文**:
  ```json
  [
    {
      "role": "system",
      "content": "你是极速意图路由引擎... \n【Active Canvas 活跃画布】:\n(无活跃任务)\n\n【Archive Memory 最近归档记忆索引】:\n(无)"
    },
    { "role": "user", "content": "看下 report.txt 并整理 Downloads 的 PDF。" }
  ]
  ```
  - **SLE 输出预期**:
  ```json
  {
    "intents": [
      { "intent_id": "req_1", "type": "NEW_TASK", "tool": "openClaw", "task_name": "读取报告", "query": "读取 report.txt" },
      { "intent_id": "req_2", "type": "NEW_TASK", "tool": "openClaw", "task_name": "归档PDF", "query": "整理 Downloads 里的 PDF" }
    ],
    "isAnswerInActiveCanvas": false,
    "isAnswerInArchiveMemory": false
  }
  ```
  *(注: 因为 `isAnswerInActiveCanvas: false`，所以直接触发正常的 SLC 响应，不强制注入画布上下文。)*
- **步骤 2: SLC Payload 预期 (生成兜底垫词)**:
  - `User`: "看下 report.txt 并整理 Downloads 的 PDF。"
  - `Assistant (应输出)`: "好的先生，正在为您读取文件并整理归档..."

### 2.2 任务执行中的并行与查询 (Preemption & Status Query)
- **触发**：后台跑着归档任务，用户同时问天气。
- **当前 Canvas 物理状态**:
  ```json
  {
    "tasks": [
      { "id": "t_01", "name": "读取报告", "status": "COMPLETED", "summary": "结论是下周二...", "is_delivered": true },
      { "id": "t_02", "name": "归档PDF", "status": "PENDING", "stage": "MOVING_FILES", "progress": "50%", "is_delivered": false }
    ]
  }
  ```
- **步骤 1: SLE (ROUTING) 意图识别的 Message 上下文**:
  `PromptAssembler` 会动态提取上述 JSON 中所有的任务放入 `【Active Canvas】` 区域。
  ```json
  [
    {
      "role": "system",
      "content": "你是极速意图路由引擎... \n【Active Canvas 活跃画布】:\n[t_01] 读取报告 (已完成): 结论是下周二...\n[t_02] 归档PDF (执行中): 阶段 MOVING_FILES, 进度 50%\n\n【Archive Memory】:\n(无)"
    },
    { "role": "user", "content": "今天上海下雨吗？归档到哪了？" }
  ]
  ```
  - **SLE 输出预期**:
  ```json
  {
    "intents": [
      { "intent_id": "req_1", "type": "NEW_TASK", "tool": "openClaw", "task_name": "查上海天气", "query": "查查上海今天天气" }
    ],
    "isAnswerInActiveCanvas": true, // 命中！发现需要结合 Canvas 中的归档进度回答
    "isAnswerInArchiveMemory": false,
    "matched_task_ids": ["t_02"] // [核心优化] 精确告诉系统，后续只把 t_02 的状态塞给 SLC (注: 该ID数组可通用指向 Active 或 Archive 记忆)
  }
  ```
- **步骤 2: SLC Payload 预期 (受到 Router 指令，注入潜意识)**:
  - `User`: "今天上海下雨吗？归档到哪了？"
  - `Assistant 注入先导潜意识`: "(潜意识: 刚才用户问了问题。目前后台长任务状态：[ID: t_02] '归档PDF' 正在 MOVING_FILES (已处理 50%)。我需要一边回答天气，一边顺口播报这个进度。)"
  - `Assistant (应输出)`: "先生，正在查天气。另外文件已移动一半了..."

### 2.3 多任务完结播报与聚合 (Broadcast Aggregation)
- **触发**：Watchdog 轮询发现两个任务(t_01 和 t_02)几乎同时 `status: COMPLETED` 且 `is_delivered: false`。主动打断并推送。
- **当前 Canvas 物理状态**:
  ```json
  {
    "tasks": [
      { "id": "t_01", "status": "COMPLETED", "summary": "报告结论是...", "is_delivered": false },
      { "id": "t_02", "status": "COMPLETED", "summary": "15个PDF已归档", "is_delivered": false }
    ]
  }
  ```
- **执行序列**：Watchdog 直接发起内部 Trigger 触发 SLC 生成，不走 SLE ROUTING。
- **SLC Payload (被 Watchdog 提取未播报的 Summary 注入)**:
  - `Assistant 影子首发潜意识`: "(潜意识: 后台主动推送了总结！画布显示 t_01 与 t_02 均已完成。我需要把以下结果合并自然地播报给先生：1. 15个PDF已归档。 2. 报告结论是xxx。必须自然，不得机械复读！)"
  - `Assistant (应输出)`: "先生，办妥了。PDF已存放好，另外刚才那份报告我看过了..."

### 2.4 短效追问命中归档 (Short-term Query hitting Archive)
- **触发**：用户追问已经被清除出 Canvas 活跃区的历史归档任务细节（如 `report.txt`）。
- **当前 Canvas 物理状态**: `{"tasks": []}` (归档清理完毕，画布恢复空白)
- **当前 DialogueMemory (长线记忆库) 状态**:
  `[{ "event": "TASK_ARCHIVED", "payload": { "id": "t_01", "name": "读取 report.txt", "summary": "核心结论是下周二架构评审, 下周四联调发布。" } }]`
- **步骤 1: SLE (ROUTING) 意图识别的 Message 上下文**:
  为了能够让 Router 得出 `isAnswerInArchiveMemory: true` 的结论，`PromptAssembler` 必须且只需提取 `DialogueMemory` 中最近的 `TASK_ARCHIVED` 事件放入 Payload：
  ```json
  [
    {
      "role": "system",
      "content": "你是极速意图路由引擎 (SLE ROUTER)。\n\n【Active Canvas 活跃画布】:\n(无活跃任务)\n\n【Archive Memory 最近归档记忆索引】:\n[Task ID: t_01] \"读取 report.txt\" (5分钟前归档)\n摘要: 核心结论是下周二架构评审, 下周四联调发布。\n\n输出JSON格式：\n{ \"intents\": [], \"isAnswerInActiveCanvas\": boolean, \"isAnswerInArchiveMemory\": boolean }"
    },
    {
      "role": "user",
      "content": "刚才报告的第二点是什么？"
    }
  ]
  ```
  - **SLE 输出预期 (成功截断外调，直接回答)**:
    ```json
    {
      "intents": [], 
      "isAnswerInActiveCanvas": false,
      "isAnswerInArchiveMemory": true, // 成功命中归档记忆索引！不要再调 read_file 了！
      "matched_task_ids": ["t_01"] // [核心优化] 告诉系统只需把 t_01 的历史回忆塞给 SLC
    }
    ```
- **步骤 2: SLC Payload 预期 (收到 Router 肯定指令后，注入归档记忆)**:
  - `User`: "刚才报告的第二点是什么？"
  - `Assistant (潜意识带入历史)`: "(潜意识: 刚才用户的提问在历史归档中找到了答案：内容是[摘要: 结论是周二评审, 周四发布]。我需要回答他而无需说我在查询。)"
  - `Assistant (应输出)`: "第二点是周四的联调发布，先生。"

### 2.5 边界异常场景与防反转 (Edge Cases & Boundaries)
- **场景 M1 - 任务强制取消与覆盖 (Cancellation & Override)**:
  - **触发**：用户在 t_02 (归档PDF) 还是 PENDING 时说："算了别归档了，直接把它们全删了。"
  - **SLE (ROUTING) 预期**：Router 必须扩展出 `CANCEL_TASK` 意图，并在 Payload 中注入画布任务 ID，借此实现长耗时任务的强行中断。
    ```json
    {
      "intents": [
        { "intent_id": "req_1", "type": "CANCEL_TASK", "target_task_id": "t_02" },
        { "intent_id": "req_2", "type": "NEW_TASK", "tool": "openClaw", "task_name": "删除PDF", "query": "删除 Downloads 里的所有 PDF" }
      ]
    }
    ```
- **场景 M2 - 任务依赖与串联 (Task Dependency)**:
  - **触发**：用户说："先读取 report.txt，然后根据里面的内容帮我发封邮件。"
  - **SLE (ROUTING) 预期**：引入 `depends_on` 机制。Orchestrator 解析时，将具备 `depends_on` 的意图置于等待队列。
  - **异常策略约定**：
    - **失败传播 (Propagation)**：若前置任务 FAILED，则关联的下游任务转为 CANCELLED，且 SLC 要主动告知：“因为第一步异常，后续动作已取消。”
    - **超时降级 (Timeout)**：如果处于等待状态的依赖节点卡死超 120s，无条件降级为 FAILED。
    ```json
    {
      "intents": [
        { "intent_id": "req_1", "type": "NEW_TASK", "tool": "openClaw", "query": "读 report.txt" },
        { "intent_id": "req_2", "type": "NEW_TASK", "tool": "openClaw", "query": "发邮件", "depends_on": "req_1" }
      ]
    }
    ```
- **场景 M3 - 极深历史追问兜底 (Deep History Beyond Archive)**:
  - **触发**：用户追问一个半小时前已彻底淘汰出 `Archive Memory` 注水窗口的任务。
  - **SLE/SLC 预期**：由于 SLE ROUTING 未能匹配 `isAnswerInArchiveMemory`，也没有触发新工具。此时将退化为空白态聊天。SLC Prompt 必须包含兜底策略：“对于超脱记忆库的碎片细节，诚恳表示当前短时记忆无法回忆，主动提议是否重新检索。”

## 3. 技术演变路线 (Tech Roadmap for 3.7)

### 3.1 Canvas 数据结构升级 (目标强类型)
- **数组化 (Array)**：将 `task_status` 升级为 `tasks[]`。明确接口契约：
  ```typescript
  interface TaskItem {
      id: string;              // 唯一任务 ID (如 "t_01")
      name: string;            // 任务名称
      status: 'PENDING' | 'READY' | 'COMPLETED' | 'FAILED' | 'CANCELLED';
      stage?: string;          // 执行子阶段 (如 "MOVING_FILES")
      progress?: string;       // 进度百分比或进度短描
      summary: string;
      direct_response?: string;  // [恢复] SLC 可直接播报的净化摘要
      extended_context?: string; // [恢复] 追问时查阅的扩展上下文
      importance_score: number;  // [恢复] Watchdog 播报分级阈值 (1-10)
      is_delivered: boolean;
      created_at: number;
      completed_at?: number;   // 完结时间戳 (供 TTL 清理)
  }
  interface CanvasState {
      env: { time: string; weather: string };
      tasks: TaskItem[];       // 核心重构区
      context: {               // [恢复] 严格强类型，供 Watchdog 依赖判断
          last_spoken_fragment: string;
          interrupted: boolean;
          last_interaction_time: number;
          is_busy: boolean;
          idle_trigger_count?: number;
      };
  }
  ```

### 3.2 SLE 意图分解协议 (Router Array Upgrade)
- `IntentRouter` 需要支持**意图阵列 (Intent Array)** 分层分发，不再是一个简单的 `needsTool: boolean`。
- **动态控制意图**：`intents` 数组内的元素分为 `NEW_TASK` (新增调起)、`CANCEL_TASK` (针对性取消)、以及具备 `depends_on` 标志的串联任务。
- **截断外调机制**：增加 `isAnswerInArchiveMemory` 字段。如果查询已被归档的任务信息，SLE 坚决不重新调用耗时的只读网络/系统工具。
- **参数调优**：为容纳多意图数组，Router 生成的 `max_tokens` 必须由 50 提升至 **300**。
- **附录 - Router Output Schema 终态契约**:
  ```json
  {
      "intents": [
          {
              "intent_id": "req_1",
              "type": "NEW_TASK" | "CANCEL_TASK" | "CLARIFY",
              "tool": "openClaw",
              "task_name": "string",
              "query": "string",
              "target_task_id": "string",
              "depends_on": "req_0",
              "message": "string"
          }
      ],
      "isAnswerInActiveCanvas": boolean,
      "isAnswerInArchiveMemory": boolean,
      "matched_task_ids": ["string"]
  }
  ```

### 3.3 I/O 拼接精调：PromptAssembler 与 潜意识 (SLC)
- **Router 输入格式化**：新增 `formatCanvasForRouting(tasks)` 函数，绝对禁止将任务的全量 JSON 塞给 Router。必须将其转为极简版人类可读摘要（例如 `[t_01] 读取报告 (已完成): 结论是...`）置于 Router 的 System Prompt 中。
- **硬盘提取 (对 SLE Router)**：为 SLE Router 注入 Active Canvas 摘要以及提取自长线回忆的 Archived Tasks Index。
- **精准投喂 (对 SLC Generator)**：基于 Router 抛出的 `matched_task_ids` 做过滤拦截。
- **潜意识升级**：`buildShadowThought(type, tasks)` 的方法签名需重构，接受 `TaskItem[]` 数组入参，而非单一字符串。
- **附录 - SLC 在多场景下的 Message 矩阵分配**:
  | 场景 | system | history | user | assistant (潜意识 Pre-fill) |
  | :--- | :--- | :--- | :--- | :--- |
  | 2.1 新发子任务 | persona+TTS规范 | 真实对话序列 | 原文 | `PROGRESS_REPORT(【空或新增标记】)` |
  | 2.2 混合并行回答 | persona+TTS规范 | 真实对话序列 | 原文 | `chat(matched_tasks 提取)` |
  | 2.3 聚合监控播报 | persona+TTS规范 | 真实对话序列 | "(隐式Trigger)" | `RESULT_DELIVERY(completed_tasks)` |
  | 2.4 深潜记忆命中 | persona+TTS规范 | 真实对话序列 | 原文 | `chat(archive_matched_tasks 提取)` |

### 3.4 Watchdog GC 守护 (冷热分离)
- **短效驻留 (Hot Retention TTL)**：任务在被置为 `COMPLETED` 并完成首轮播报后，保留约 2 分钟或 3-5 轮对话在 Canvas，防范用户高频追问。
- **静默归档与物理清理 (Cold Archiving & GC)**：
  - 触发时机：TTL 到期或用户明确转移话题。
  - 逻辑：将精简的 `summary` 封装为 `TASK_ARCHIVED` 事件存入长线记忆库 (`DialogueMemory`)。
  - 从 Canvas 的活跃 `tasks[]` 数组中彻底删除，如果是文件级操作，触发 `cleanup_hook` 做物理擦除。

### 3.5 AgentOrchestrator 并发分流与 DECIDING 隔离
- **混合编排策略**：当产生如 2.2 场景的并行意图时 (`intents[]` 非空且 `isAnswerInActiveCanvas=true`)，Orchestrator 将并发派发新任务至 Executor 进行逻辑推理，同步由另一路根据 `matched_task_ids` 组装 SLC 潜意识兜底回复。实现“一边答疑、一边接新活儿”的双路协同。
- **异步与物理中断 (CANCEL_TASK 的闭环)**：
  - Orchestrator 将被杀任务标为 `CANCELLED`。
  - `ToolResultHandler` 必须全局维护一个 `Map<taskId, AbortController>`。侦测到取消信号，调用 `abort()` 触发底层代码/流进程硬中断。
- **DECIDING 上下文注入重定义 (核心盲区修复)**：
  - **隔离快照**：之前 DECIDING 塞的是全局画布 JSON，多线程下绝对禁止污染！现在必须**仅提取当前特定 intent 对应的独立 TaskItem 切片**塞给底层 Agent。
  - **Intent Hint 移花接木**：`SLE_ACTION_PROTOCOL` 的线索必须源自 Router 精准指派给该路的 `intents[i].query`。
  - **强绑定防串区**：工具输出的摘要（`response`）必须带上回执 `taskId` 盖章写入 Canvas，坚决防范各个工具把同伴画布写炸的情况。

### 3.6 DialogueMemory 事件体系支持
- **当前限制**：原 `.jsonl` 存储仅支持 `{role: string, content: string}` 的非结构化对话。
- **升级要求**：新增 `logEvent(callId, eventType, payload)` 接口。允许将超时淘汰的 `TASK_ARCHIVED` 作为强类型的 JSON 块追加到记忆文件，确立“长线工作簿”。

## 4. 关键挑战与规避 (Challenges)
1. **意图指代消解 (Pronoun Resolution)**：用户说“那个别弄了”，如何对应到特定的 Shell 进程？
   - **解决方向**：利用注入了 Active Canvas 的 Router System Prompt 引导大模型根据就近性（Recency Heuristic）消解歧义（优先归属最后一个 PENDING 任务）。如果存在多个高度相似任务引发歧义，允许 Router 输出 `{"type": "CLARIFY", "message": "您是指..."}` 意图拦截。
2. **上下文污染**：多任务执行的中间文本可能导致 SLC 回复过于啰嗦，需要更精准的 `summary`。
3. **性能开销**：控制多路 Agent 决策时对 Token 消耗与延迟的平衡。
